## 식별자와 키
<details>
<summary>식별자란?</summary>

> 엔티티에서 각 인스턴스를 유일하게 구분하는 속성, DB에서는 키(PK)라고 한다.

- 엔티티 : 인스턴스들의 집합
    - 집합에서 인스턴스를 어떻게 식별하지?
    - 그것을 가능하게 해주는게 `식별자`
- 조건 
    1. 유일성 : 값이 겹치지 않아야 한다.
    2. 최소성 : 식별자를 이루는 요소를 더 줄이면, 유일성이 깨진다. 
        - 즉, 꼭 필요한 속성만으로 구성
</details>

<details>
<summary>슈퍼키/후보키 차이는?</summary>

> 최소성 허용 여부가 다르다.
- 슈퍼키 : 유일성 O / 최소성 X
    - 즉, 유일하게 구분만 되면 군더더기가 붙어도 슈퍼키가 된다.
- 후보키 : 유일성 O / 최소성 O
    - 유일성과 최소성을 만족하여, PK(키)의 후보라는 의미
</details>

<details>
<summary>대체키란?</summary>

> 후보키에서 기본키를 뺀 것
- 후보키 = 기본키 + 대체키
- 대체키는 보통 DB에서 UNIQUE제약으로 일단 살려둔다.
</details>

<details>
<summary>자연키와 인공키의 차이는?</summary>

> 자연키는 의미가 있는 속성을 키로 사용하는 것이고, 인공키는 의미가 없는 숫자를 키로 사용하는 것이다.

- 자연키
    - 업무상 의미가 있는 값
        - ex, 주민번호, 이메일, 학번
    - 데이터 자체에 뜻이 있다.
- 인공키
    - 시스템이 인위적으로 만든 값
        - ex, auto_increment ID, UUID
    - 그냥 일련번호, 뜻 없음

- 최근엔 ORM등을 많이 사용하고, 자연키는 문제가 생길 위험을 내재하기 때문에 인공키를 많이 선호한다.
    - 그러나, 인공키를 사용하더라도 자연키는 UNIQUE 제약으로 살려두는게 좋다.
        - 나중가서 제약을 추가하기는 어렵기 떄문
</details>

<details>
<summary>인공키의 장점?</summary>

> 유일성 보장 / 편의성 / 변경할 일 없음 / 자식이 많을 때 유용

1. 유일성 보장 : 자연키만으로는 유일성이 지켜지지 않을 때, 인공키는 무조건 지킨다.
2. 편의성 : 자연키가 길어서 JOIN이 복잡할 때 인공키는 간단하다/
3. 변경할 일 없음 : 자연키는 값이 바뀔 수 있는데 인공키는 없다.
4. 자식이 많을 때 : 인공키는 하나이기 때문에 키를 상속시켜도 좋다.
</details>

<details>
<summary>인공키 UUID 단점?</summary>

> 16바이트를 차지 / v7전까지는 시간 순서 관리가 불가했음

1. 16바이트 차지
    - UUID는 128bit로 16바이트다.
    - 이것을 가지는 것 자체도 크지만, 상속하는 경우 자식 테이블들도 다 16바이트를 갖는다는 것은 매우 부담
2. 시간 순서
    - UUID는 ID와 다르게 sequential하지 않다. (이것 자체가 장점이기도 함)
        - 장점인 이유 : sequential하다는 것 자체가 정보를 갖고 있음, 내가 몇 번째 순선지, 등
            - UUID는 이것을 숨길 수 있다.
    - 따라서 시간 순서로 정렬이 필요한 경우, 다른 컬럼에 index를 만들고 정렬을 진행해야 한다.
        - ID였다면 따로 인덱스를 만들 필요도 없는데 , UUI는 불편하다.
    - But, v7부터는 앞 바이트 일부가 순서를 나타내게 되어서 괜찮아졌다.
</details>

<details>
<summary>PK 선정 기준</summary>

1. NOT NULL 
2. 불변성 : 잘 안 변하는 거
3. 단순성 : 짧고 단순한거 (JOIN에 유리)
4. 대표성 : 이 엔티티를 가장 잘 대표하는 것
</details>

<details>
<summary>외래키란?</summary>

> 다른 엔티티의 기본키(PK)를 참조하는 속성
>> 관계(1:N)를 테이블에서 표현하는 수단
- FK를 통해 참조 무결성을 지킬 수 있다.
    - FK 제약조건을 걸어두면, 레코드 삽입마다 PK를 검사한다.
        - 이로 인해, 제약 조건이 추가되면 성능에 약간 악영향이 생긴다.
            - 최근엔 무조건 빠른걸 선호하기 때문에, FK제약조건을 빼고 테이블을 만드는 경우도 많다.
            - 빼고 만들더라도 인덱스는 만든다.
            - 인덱스가 있어야 JOIN했을 때 성능이 O(N+M)이 된다.
</details>

<details>
<summary>참조 무결성이란?</summary>

> FK 값은 반드시 참조 대상 테이블에 존재하는 값이거나 NULL이어야 한다는 규칙
- 존재하지 않는 회원의 주문 같은 고아 레코드를 막는 목적이다.
- FK제약이 걸리면 DBMS가 다음과 같이 검사한다.
    - 자식 : INSERT할 때 부모 PK 검사
    - 부모 : 자식 테이블이 참조 중인 값을 DELETE/UPDATE할 때 지정된 옵션에 따라 처리
</details>

<details>
<summary>보조 식별자란?</summary>

> 기본키 말고 인스턴스를 식별할 수 있는 또 다른 속성
- 다른 엔티티와 연결고리 역할이 필요할 때 정의한다.
    - 실무에서는 UNIQUE INDEX로 구현되는 경우가 많다.
        - 항상 UNIQUE INDEX는 아님
</details>

<details>
<summary>UNIQUE INDEX란?</summary>

> UNIQUE + INDEX를 해준다.

- UNIQUE 제약을 걸면 내부적으로 UNIQUE INDEX를 자동 생성해서 구현한다.
    - 굳이 뭐 CREATE UNIQUE INDEX 이렇게 안 해도 괜찮다.
</details>

<details>
<summary>자연키가 깨지는 순간</summary>

> 논리적으로는 맞아도, 현실에서 문제가 생길 때가 많다.
1. 유일성 실패 : 정말 우연치 않게 자연키가 겹칠 수 있다.
2. 너무 길고 복잡 : 키 여러 개로 복합키를 이루는 경우 이 모든 키를 자식테이블에 끌고 다님
    - JOIN 조건이 길어짐
3. 값이 바뀜 : 이메일, 전화번호, 학번처럼 나중에 변경될 수 있음
    - PK 하나 바뀌면 참조하던 FK가 모두 변경되어야 한다.
</details>

## 정규화


<details>
<summary>데이터 중복이 나쁜 이유</summary>

> 반복되면, 언젠간 달라져서 무엇이 진짠지 알 수 없게 된다.
>> \+ 저장공간 낭비 \+ 이상현상이 발생하게 된다.
</details>

<details>
<summary>이상현상이란?</summary>

> 데이터 중복으로 인해 테이블을 조작할 떄 생기는 비합리적인 현상
- 종류
    1. 삽입 이상
        - 넣고 싶지 않은 데이터까지 억지로 넣어야 하는 문제
        - 필요없는 컬럼의 값까지 넣어야 값이 삽입되어 가짜 값을 넣거나 등록이 안되는 경우
    2. 갱신 이상
        - 하나를 바꿀라면 여러 행을 동시에 고쳐야 하는 문제
        - 이때, 실수로 놓치는게 생기면 데이터 정합성이 깨진다.
    3. 삭제 이상
        - 지우고 싶지 않은 정보까지 함께 삭제되는 문제
        - 하나 행에 여러 주제가 섞여있는 경우 수강 취소할라했는데 과학이라는 과목 자체가 사라질 수도 있음
</details>

<details>
<summary>함수 종속성이란?</summary>

> X->Y : X를 알면 Y가 결정된다. / Y는 X에 종속된다. / X가 Y를 결정한다.
- X : 결정자 / Y : 종속자
- Y는 X에 종속되기 때문에 Y가 달라지면, X가 달라진다.
    - but, X가 달라진다고 Y가 달라지지는 않는다.
        - ex, 학번 -> 학생이름 
            - 학번이 달라져도 학생 이름은 같을 수 있다. (동명이인)
            - 학생 이름이 달라지면, 학번이 달라진다.
</details>


<details>
<summary>완전 함수적 종속이란?</summary>

> 키 전체가 있어야 값이 정해지는 것
</details>

<details>
<summary>부분 함수적 종속이란?</summary>

> 키의 일부만 있어도 값이 정해지는 것
</details>

<details>
<summary>이행 함수적 종속이란?</summary>

> 키가 아닌 것에 의해 값이 정해지는 것
</details>

<details>
<summary>제1정규형이란?</summary>

> 모든 속성은 원자값을 가져야 한다. (반복 그룹 제거)
- 한 칸의 여러 값을 행으로 풀어주는 것
    - 이로 인해 부분 함수적 종속이 발생할 수 있다.
</details>

<details>
<summary>제2정규형이란?</summary>

> 복합키의 일부에만 종속되는 속성을 없앤다.
- 부분 종속 제거
    - 이로 인해 이행 종속이 생길 수 있다.
</details>

<details>
<summary>제3정규형이란?</summary>

> 키가 아닌 속성끼리 종속되면 안 된다.
- 이행 종속 제거
    - 이때, 주의할 점
        - 키가 아닌 속성이 키를 결정하는 것은 3정규형을 만족하는 것으로 본다.
</details>

<details>
<summary>BCNF?</summary>

> 모든 결정자는 후보키여야 한다.
- 키가 아닌 속성이 키의 결정자인 경우, 3정규형을 만족하는 것으로 보지만, BCNF는 만족하지 못하는 것으로 본다.
    - 이를 해결하기 위해 한 번 더 분리하는 것이 BCNF다.
</details>

<details>
<summary>정규화의 이점?</summary>

1. 응집도 높은 엔티티가 도출된다.
2. 중복이 최소화되어 이상현상이 사라지고, 저장효율이 좋아진다.
3. 무결성이 높아진다. (정합성이 보장됨)
4. 주 식별자 정확성이 높아진다. (올바른 PK가 드러남)
5. 확장성이 높아진다. (업무 변경 시 영향 범위가 명확하고 작음)
</details>

## 물리 모델링

<details>
<summary>물리 데이터 모델링이란?</summary>

> 논리 데이터 모델을 특정 DBMS에 맞는 물리적 요소로 적용하는 과정
</details>

<details>
<summary>물리 모델링 절차</summary>

1. 테이블 매핑 (엔티티 -> 테이블, 단위랑 우선순위 결정)
2. 컬럼 매핑 (식별자 -> PK, 일반 속성 -> 컬럼, 관계 -> FK)
3. 상세화 (무결성, 인덱스, 반정규화, 저장기법 결정)
</details>

<details>
<summary>서브타입 처리 방안 3가지</summary>

1. Roll-Up : 하나로 합치기
    > 슈퍼타입 이름으로 테이블 하나, 구분자 컬럼으로 서브타입 구분
2. Roll-Down : 다 쪼개기
    > 서브타입마다 테이블 하나씩 생성, 공통 속성들도 모두 서브타입 테이블에 포함
3. Hacksaw : 슈퍼타입 하나, 서브타입 각각 
    > 슈퍼타입은 하나의 슈퍼타입 테이블로 관리, 각 서브타입별로 테이블 생성
</details>
<details>

<summary>Roll Up의 장단점은?/언제 사용?</summary>

> 서브 타입끼리 비슷하고, 전체 조회하는 일이 많을 때 사용
- 특징 : 서브타입별 전용 컬럼은 NULL허용할 수 밖에 없음
    - 장점 :
        - 조회 간단 / 조인이 줄어 속도 빠름
        - 복잡한 로직을 하나의 SQL로 처리
        - 전체 조회 간단
        - 새 서브타입 추가 시 값만 추가
    - 단점 :
        - 서브타입별 NOT NULL 제약이나 기본값을 일관되게 주기 어렵다.
        - 저장공간 증가 가능
        - 항상 구분자로 걸러야 한다.
        - 일부 서브타입만 보고 싶어도 FULL SCAN한다.
</details>

<details>
<summary>Roll Down의 장단점은?/언제 사용?</summary>

> 서브타입별로 조회/처리할 일이 많을 떄

- 장점 :
    - 서브타입별 조회 시 구분 불필요
    - 가져올 컬럼이 명확
    - NOT NULL, 기본값을 제대로 지정할 수 있음
    - 테이블 크기가 작아서 FULL Scan에 유리
- 단점 :
    - 둘을 같이 보려면 UNION 필요
        - 성능 저하
    - 통합 SQL 작성 어려움
    - 전체 식별자 유지 관리 어려움
    - 합친 VIEW는 조회만 가능하다.
</details>

<details>
<summary>Hacksaw의 장단점은?/언제 사용?</summary>

> 공통(슈퍼) 트랜잭션이 많고, 서브타입 처리도 독립적으로 자주 발생하는 경우

- 장점
    1. 통합/분리의 장점을 동시 수용
    2. 슈퍼타입만 조회 시 SQL이 간단
    3. 컬럼 중복 없음
    4. 슈퍼/서브 업무 중요도가 대등할 때 적합
- 단점
    1. 같이 처리하려면 항상 조인 발생
    2. 전체 통합 조회 많으면 과다 조인
    3. SQL이 한층 복잡해질 수 있음
    4. 슈퍼 <-> 서브 데이터 동기화 필요
</details>
    
<details>
<summary>컬럼 NULL 허용은 왜 신중하게 해야할까?</summary>

> 데이터의 활용 형태에 따라 결정해야 한다.
- NULL은 버그가 자주 생기는 원인이기 때문이다.
    - 예를 들어, NOT IN 비교를 하는 경우, NULL은 NOT IN에서 없음으로 표기되지 않기 때문에 제외된다.
</details>

<details>
<summary>실체 무결성이란?</summary>

> 모든 테이블은 PK를 가져야 하고, PK값은 NULL이거나 중복일 수 없다는 규칙

- 이유
    - PK는 각 레코드를 식별하는 수단인데, NULL이거나 중복이거나 없으면 행을 유일하게 특정할 수 없다.
- 구현
    - PRIMARY KEY 제약으로 보장한다.
</details>

<details>
<summary>도메인 무결성이란?</summary>

> 각 컬럼의 값은 정의된 도메인 안에 있어야 한다는 것

- 도메인: 데이터 타입 + 길이 + NULL 허용 여부 + 허용 값 범위/목록
    - 삽입 시, 도메인 값에 정의된 값만 잘 들어가도록 하는 것

- 구현 수단
    - NOT NULL / CHECK / DEFAULT / ENUM / 코드 테이블
</details>

<details>
<summary>사용자 정의 무결성이란?</summary>

> 업무 요구사항에 나오는 규칙

- 예시
    - "주문 취소는 배송 시작 전에만 가능하다" — 상태 전이 규칙
    - "대출 총액은 고객 신용한도를 초과할 수 없다" — 여러 행/테이블에 걸친 집계 조건
    - "미성년 회원은 성인 인증 상품을 주문할 수 없다" — 테이블 간 조합 조건

- 구현 수단
    - 트리거 / 저장 프로시저 / 애플리케이션 코드(서비스 로직)
</details>

<details>
<summary>FK 삭제 옵션의 차이, 최근 삭제 트렌드</summary>

> 최근에는 Soft Delete를 많이 사용한다. (저장공간이 싸지고, 데이터가 어떻게 될지 모르므로)

- Soft Delete : 테이블에 is_deleted같은 구분 컬럼을 하나 놓고, 해당 컬럼의 값을 바꾸는 형식으로 제거처리하는 것
- FK 삭제 옵션
    - CASCADE : 부모 삭제 시 자식도 삭제
    - SET NULL : 부모 삭제 시 자식 FK는 NULL
    - RESTRICT : 참조되는 부모는 삭제 불가
</details>

<details>
<summary>반정규화란?</summary>

> 조회 성능을 위해 일부러 중복을 허용하는 것
- 조회 성능을 얻는 대신, 정합성 관리 비용을 떠안게 된다.
    - 반정규화가 진행될 수록 어플리케이션 코드가 복잡해진다.
</details>

<details>
<summary>반정규화 종류</summary>

1. 집계 테이블 : GROUP BY 결과를 미리 계산해서 저장
2. 진행 테이블 : 흩어진 상태/진행 정보를 별도 테이블로 통합 관리
3. 테이블 통합 : 자주 같이 쓰는 테이블을 하나로 (1:1 등)
4. 테이블 분할 : 자주 쓰는 컬럼/행만 떼어내기 (수직,수평)
5. 컬럼 반정규화 : 부모/자식에 컬럼을 복사해 조인 제거
</details>

<details>
<summary>인덱스 설계</summary>

- PK/FK엔 인덱스가 자동으로 생성된다.
- 자주 조회하는 컬럼에 인덱스를 사용하자(WHERE)
    - 쿼리시 인덱스를 타는지 쿼리 플랜과 비쥬얼 플랜을 확인하자. (이건 항상 확인해야 한다)
- 자주 같이 사용되는 컬럼은 결합 인덱스로 묶자.
    - 결합 인덱스는 순서가 중요하다.
- 분포도가 낮은 컬럼일 수록 효과적이다.
    - ex, 불균형 성비 속성에 인덱스를 타면, FullScan보다 효율이 떨어진다.
        - 일반적으로 전체의 20%이상 조회하게 된다면 인덱스를 제거한다.
    - FullScan은 테이블 블록을 순차적으로 읽는 것
        - 디스크 입장에서 효율적인 패턴
    - 인덱스를 통한 접근은 테이블 블록이 흩어져 있어서 랜덤 I/O가 발생함
        - 쉽게 말하면, 요소 1,2,3,4,5,6이 테이블 a,b에 3개씩있고, 인덱스가 1-4-2-3 이렇게 진행된다고 할 때, a갔다가 b갔다가 a갔다가 이렇게 해야 돼서 안 좋다는 것이다.
    - 하지만, 커버링 인덱스일 때는 index-only scan이라 인덱스가 또 좋다.    
        > 인덱스가 느려지는 것은 인덱스 탐색이 아닌, 인덱스->테이블로 넘어가는 랜덤 액세스
        - 커버링 인덱스 : Index에 있는 Column이 쿼리가 요구하는 컬럼을 커버하는 거
            ```sql
            CREATE INDEX ix_주문 ON 주문(회원ID, 주문일시, 주문상태);

            SELECT 주문일시, 주문상태
            FROM 주문
            WHERE 회원ID = 100;
            ```
</details>

<details>
<summary>대상의 성격에 따라 엔터티를 3개로 분류하면? </summary>

    - ex, 사원, 자동차 등
2. 개념 엔터티
    - 물리적 형태는 없지만, 관리해야 하는 개념
    - ex, 부서, 회원 등급, 직책 등
3. 사건 엔터티
    - 특정 행위나 사건의 결과
    - ex, 주문, 결제 등
</details>

<details>
<summary>발생 순서에 따라 3개로 분류하면? </summary>

1. 기본 엔터티
    - 원래부터 업무에 있는 대상, 의존 X
    - ex, 회원, 상품, 부서 등
2. 중심 엔터티
    - 기본 엔터티들이 만나서 업무의 중심 사건이 됨
    - ex, 주문, 결제 등
3. 행위 엔터티
    - 중심 엔터티에서 반복적으로 쌓이는 상세/이력
    - ex, 주문상세, 결제내역, 배송이력 등
</details>

    - 즉, 이름만 보고 판단하면 안 되고 업무의 성격(업무에서 무엇을 표현하는지)를 먼저 확인해야 한다.
</details>

    - 코드를 의미별로 분리해서 관리해야 한다.

</details>

<details>
<summary>엔터티와 테이블의 차이는?</summary>

> 엔터티 : 무엇을 관리할 것인가에 대한 개념 답변 / 테이블 : 엔터티를 어떻게 저장할 것인가에 대한 물리적 구현 답변

- 엔터티 : 논리적 설계 개념
    - 업무에서 관리해야 할 데이터의 집합을 개념적으로 정의한 것
        - 주문/회원 등 **업무 관점**에서 식별되고, **DBMS에 독립적**이다.
    - 속성과 식별자를 가진다.

- 테이블 : 물리적 구현 결과물
    - 엔터티를 특정 DBMS에 맞게 구현한 저장 구조
        - Column / PK,FK 제약 / 인덱스 / 데이터 타입 등 물리적 요소를 가짐

- 물리 설계 과정에서 변환되기 때문에 둘은 반드시 1:1로 대응되지 않는다.
    - EX 1. 수퍼타입/서브타입 모델
        - 하나로 통합하거나 서브타입별로 분리하면서 달라진다.
    - EX 2. M:N 관계
        - 엔터티에 없던 교차 연결 테이블이 물리적 구현에서 추가된다.
    - EX 3. 반정규화
        - 여러 엔터티가 하나의 테이블로 합쳐지거나 컬럼이 중복되게 된다.
    - EX 4. 대용량 파티셔닝 / 수직 분할
        - 하나의 엔터티가 여러 테이블로 나누어지게 됨


</details>
<details>
<summary>상태와 상태 이력을 분리하는 이유는?</summary>

값이 계속 바뀌는 속성을 UPDATE로만 관리하게 되면, 과거 값을 알 수 없다.
따라서 상태 이력(변경 기록)을 별도의 이력 엔티티로 분리하여 관리한다.

- 분리하는 이유
    1. 조회 성능
        - 현재 상태를 따로 관리하지 않는다면, 이력 테이블에서 MAX로 조회해야 하는데, 이 쿼리는 본 테이블에서 현재 상태 관리하는 것에 비해 매우 느리다.
        - 따라서 분리하면, 현재 상태 조회를 빠르게 할 수 있다.
    2. 데이터 증가 패턴이 다르기 때문(저장 공간 절약)
        - 본 데이터에 비해 상태는 더 자주 변경된다. (ex, 주문이라하면, 주문 1건 당 -> 주문 접수 / 출고 / 배송 / 완료 이렇게 4개가 나온다)
        - 이렇게 증가속도가 다른 데이터를 한 테이블에서 관리하면, 테이블이 쓸데 없이 비대해지므로 분리한다.
    3. 감사(Audit)와 추적
        - 언제 누가 왜 상태를 바꿨는지 이력을 통해 추적할 수 있다.
    4. 분석 가능
        - 프로세스 시간 분석 등을 위해서는 이력 데이터가 있어야 가능하기 때문이다.
    5. 관리 정책 분리
        - 이력이 무한히 쌓이기 때문에, 파티셔닝/아카이빙/보존기간 등의 정책을 본 테이블과 독립적으로 적용할 수 있게 된다.

- 트레이드 오프 존재
    - 상태와 이력 최신이 불일치 할 수 있기에, 상태 변경 시 두 테이블을 하나의 트랜잭션으로 갱신해야 한다. (정합성을 위해)
</details>

<details>
<summary>헤더와 상세 구조를 나누는 이유는?</summary>

주문과 상품처럼 한 건의 거래 안에 여러 반복되는 그룹이 존재할 떄 분리한다.
공통 정보(헤더)와 반복 정보(상세)를 분리하여 관리한다. 
> ex)주문헤더(주문ID, 회원ID, 주문일시, 배송지, 결제수단, 총액) 주문상세(주문ID, 주문순번, 상품ID, 수량, 단가)

- 나누는 이유
    1. 제 1정규형 만족 가능
        - 한 행에 여러 상품을 작성하거나 컬럼을 반복하면 제 1정규화를 위반하게 된다.
        - 또한, 상품 수가 Column 수를 초과할 때 저장이 안 되고, 상품 ID로 주문 검색같은 쿼리도 불가능해진다.
    2. 중복 제거와 갱신 이상 방지
        - 분리하지 않고 상품별로 행을 만들면, 주문일시, 배송지 등 공통 정보가 상품 수만큼 중복된다.
            - 중복되면 또 다시 이상문제가 생기기 때문에 나눠야 한다.
    3. 데이터의 성격이 다름
        - 헤더와 상세가 다루는 성격이 다르기에 종속되는 대상이 다르다.
            - 따라서 엔터티도 다르게 해야 한다.
    4. 확장성
        - 분리하면, 주문에 상품이 몇 개가 담기든 상세 행만 추가하면 되어 확장에 유리하다.
</details>

<details>
<summary>M:N 관계를 연결 엔터티로 푸는 방법은?</summary>

- 관계형 DB는 FK로 1:N관계만 직접 표현할 수 있어 M:N 관계를 테이블로 바로 구현할 수 없다.
    - 예를 들어 학생 한 명이 여러 과목을 수강하고, 과목 하나에 여러 학생이 있을 때, 어느쪽에 FK를 두어도 반복 그룹이 생길 수 밖에 없다.
    - 이럴 때 연결 엔터티로 분리한다.
- 방식
    1. M:N 관계 확인
    2. 관계를 엔터티로 만듦 (수강, 주문 상세 등)
    3. 양쪽 부모의 PK를 FK로 포함
    4. PK를 정함
        1. 복합키 : 두 FK를 PK로
        2. 인공키(대체키) : 별도의 ID 삽입
- 부수효과
    - 관계 자체에 속하는 속성을 저장할 수 있게 된다.(주문 상세의 수량 등)
</details>

<details>
<summary>강한 엔터티와 약한 엔터티의 차이는?</summary>

> 독립적으로 존재하고 식별될 수 있는가의 차이
- 강한과 약한은 식별/비식별 관계와 다르다. (강/약은 엔터티 성질)

### 강한 엔터티
- 존재 : 다른 엔터티 없이 독립적으로 존재
- 식별자 : 자기 PK 구성
- 관계 : 없음
    - 비식별 관계거나, 부모 관계가 아예 없음
- ERD 표기 : 단일 실선 사각형
- 부모 삭제 시 : 영향 없음

### 약한 엔터티
- 존재 : 소유(부모)엔터티가 있어야만 존재
- 식별자 : 부모 PK(자기 FK) + 자신의 부분키로 PK구성
- 관계 : 부모와 식별 관계
- ERD 표기 : 이중선 사각형, 이중선(관계 표현)
- 부모 삭제 시 : 함께 삭제되거나 존재 의미 상실 (ON CASCADE)
    - 근데 최근엔 soft deletion 방식으로 함

</details>

        - 이 명령어를 입력하면 순서대로 Access Key -> Secret Key -> Region -> Format 설정한다.
            - 처음 두 개는 우리가 [IAM USER Access 정보](#iam-user-access-정보)에서 받은걸 쓰면 되고, Region과 Format은 자유이나, 적절히 상황에 맞춰쓰자

        ```bash
        vim ~/.aws/credentials

        [default]
        aws_access_key_id = AKIA...
        aws_secret_access_key = 2ueKH...
        ```
        - 계정을 추가한다면, 동일하게 \[{user_name}\]으로 추가하면 된다.
            - 추가한 계정으로 로그인할 때는 추가 parameter를 전달해서 이용하면 된다.
        ```bash
        vim ~/.aws/config

        # 위에처럼 쓰면 아래처럼 보일 것이다. 아래를 각각 적절히 설정하자.
        [default]
        region = ap-northeast-2
        output = json
        ```
    
## EC2

    - 새 키 페어 생성
    - 안전한 곳에 저장하기
6. 네트워크 설정
    - 편집 클릭
    - 퍼블릭 IP 자동 할당 켜기 (외부에서 접속해야 하는 경우)
    - 보안 그룹 생성
        - 인바운드 보안 그룹 규칙 : 외부에서 접속하는거
            - ssh : 22
            - HTTP : 80
            - HTTPS : 443
            - 접속 IP도 제한하고 싶으면 CIDR맞춰서 작성하기
            - 다른 포트를 열어야 하면, 
                1. 사용자 지정 TCP 설정
                2. 포트범위에 원하는 포트 번호 작성
                3. 소스 유형으로 원하는 CIDR 설정

## ECR

## IAM USER 권한 추가
> ECR 권한 정책을 추가해주는 방법
1. IAM 접속
2. 사용자 탭에서 추가하고 싶은 계정 선택
3. 권한 탭 -> 권한 추가 -> 직접 정책 연결
4. AmazonEC2ContainerRegistryPowerUser 입력 후 체크 -> 권한 추가
- 이러면, ECR에 해당 유저가 pull/push할 수 있게 된다.
- PowerUser : 이미지 조회/푸시/리포지토리 생성 가능, 삭제 불가능

## ECR Push 방법

1. AWS CLI 설정이 되어있는지 확인한다.
    ```bash
    aws configure
    # 또는
    cat ~/.aws/credentials 
    ```
2. ECR 로그인
    ```bash
    aws ecr get-login-password --region <리전_코드> | docker login --username AWS --password-stdin <AWS_계정_ID>.dkr.ecr.<리전_코드>.amazonaws.com
    ```
    - 리전 코드는 높은 확률로 `ap-northeast-2`(서울)
3. 이미지 빌드
    > 그냥 빌드하면 되는데, Mac인 경우에는 `--platform linux/amd64`를 붙여서 빌드해야 안 깨진다.
4. ECR 주소로 태깅
    > Docker 태그 규칙이 레지스트리/소유자/태그 여서 이거 보고 업로드하니까 ECR 태그를 추가해줘야 한다.
    ```bash
    # docker tag를 이용하여 새로운 태그 추가.
    docker tag image이름 {ECR_URI}/{user_name}/{tag}
    ```
5. push
    > 아까 태그한 이름 그대로 push
    ```bash
    docker push {ECR_URI}/{user_name}/{tag}
    ```

## ECR Pull 방법
1. AWS CLI 설정
2. AWS ECR 로그인
    ```bash
    aws ecr get-login-password --region <리전_코드> | docker login --username AWS --password-stdin <AWS_계정_ID>.dkr.ecr.<리전_코드>.amazonaws.com
    ```
    - 리전 코드는 높은 확률로 `ap-northeast-2`(서울)
3. docker run으로 해당 이름의 이미지 가져오게 실행
    > 우리가 Push한 이미지
    ```bash
    sudo docker run {ECR_URI}/{user_name}/{tag}
    ```
