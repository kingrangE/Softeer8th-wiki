# Big Data
- 특징(3V)(and 관계가 아닌 or 관계)
    1. High Volume : 양이 많다.
        - 기존 DB나 단일 서버로 저장/처리하기 어려운 TB,PB 급
    2. High Velocity : 데이터가 쌓이는 속도
        - 데이터가 생성되고 유입되는 속도가 빠르고, 처리 또한 실시간, 준실시간으로 요구된다.
        - 데이터 자체가 엄청 빠르게 생성되지 않아도 그 양이 내가 처리하는 속도에 비해 빠르게 생겨난다면 High Velocity다.
            - 이때, 내 처리를 넘어가면 어떻게 처리할지 또 고민의 영역
    3. High Variety : 데이터의 형태가 다양함
        - 정형 데이터 뿐만 아니라 반정형 비정형 모두 있음
        - 정형을 잘 다루는 것이 RDB였으나 현재는 다양해지며 다른 솔루션이 생기는 중
- 목적
    - 데이터 처리 자체가 목적이 아닌, 아래의 가치를 얻는 것이 목적
    1. enhanced insight
        - 데이터에서 패턴과 의미를 발견
    2. decision making
        - 직관이 아닌 데이터 기반 의사결정
    3. process automation
        - 사람 개입 없이 데이터가 직접 행동 유발 (ex, 추천 시스템, 자동 사기 차단)
    > 지금은 AI로 다 바뀌었다. AI Product을 위한 데이터 처리, But 내재된 목적은 결국 동일

### 5Vs of Data
> 위에 3V에 Veracity(믿을 수 있는가?)와 Value(왜 하는가?)의 2가지 가치를 더한 것
- Veracity : 데이터의 품질과 신뢰도 문제
    - 데이터 중 `불완전`하거나 `중복`되거나 `오류`가 있거나 `편향`된 데이터가 존재한다.
    - 이러한 데이터는 정제와 검증 과정이 필수라는 의미
    - Garbage In, Garbage Out
- Value : 데이터의 가치가 무엇인가의 문제
    - 수집,처리된 데이터에서 실제 `비즈니스적`, `사회적 가치`를 뽑아낼 수 있는가의 문제
    - 원시 데이터 자체는 `가치 밀도가 낮은 경우`가 많음
        - 분석을 통해 **통찰, 의사결정, 자동화로 연결해야 가치**가 생긴다.

# Parallel Processing

### Linear와 Parallel의 가장 큰 차이
- Parallel은 수십만 개를 동시에 돌려서 하나가 죽어도 자기만 다시 실행하면 된다.
    - 즉, 모두가 independent
    - 만약 Linear였다면, 매우 큰 Task에서 터지지 않도록 매우 고민하면서 제작해야 한다.
### Embarrassingly Parallel
> 가장 쉬운 형태의 병렬화
- 조건
    1. 모든 sub-problems가 계산 시작전 미리 정의된다.
        - 작업을 실행하며 동적으로 새 작업이 생성되는 것이 아니라, **시작하기 전, 완전히 분리 가능**
    2. 모든 sub-solutions가 각자 독립적인 메모리 공간에 저장된다.
        - 변수, 배열 원소와 같은 메모리 공간
        - 여러 프로세서가 같은 메모리를 동시에 쓰려고 하면 race condition이 생겨서, lock이나 동기화가 필요해진다.
            - 겹치지 않으면 걱정 없이 편하게
    3. sub-solutions 계산이 완전히 독립적
        - 각 프로세서들이 실행 중에 통신하거나 기다릴 필요없이 각자 동작
        - 병렬 컴퓨팅에서 성능을 낮추는 핵심 요인이 **프로세서 간 통신**과 **동기화** 비용
            - 3번이 보장되면, 선형적으로 속도가 빨라진다.
    4. 계산은 독립적이나, 약간의 초기화나 마지막 통신이 필요한 경우
        - 대부분 이런 경우다.
        - **nearly embarrassingly parallel**이라고 부른다.

### Data Parallelism으로 생각하기
- 우리는 보통 Task Parallelism으로 생각한다.
    - Task Parallelism
        - 서로 다른 종류의 일을 여러 프로세스/스레드에 나눠주는 방식
            - 즉, **누가 일을 맡게 할까?를 설계**한다.
            - 이러한 이유로 각 프로세스가 하는 일 자체가 다르다.
            - 각 프로세스가 하는 일이 다르니, 프로세스간 통신이 필수적이다. 연결되기 때문
        - 파이프라인 구조라고 생각하면 된다.

- 그래서 우리(DE)는 Data Parallelism으로 생각해야 한다.
    - Data Parallelism
        - 데이터를 기준으로 생각하는 것이다.
        - 관점의 중심을 데이터로 둔다. 보통은 규칙적으로 구조화된 배열을 다룬다.
        - 데이터를 Chunk단위로 겹치지 않게 잘라서 독립적으로 처리한다.
            -  조각들 사이에 **의존성이 없어야 데이터 병렬화가 성립**된다. 따라서 **어떻게 잘라도 괜찮은지를 검증**해야 한다.
- 요약하자면
    - Task Parallelism : 서로 다른 코드를 병렬 실행
    - Data Parallelism : 서로 다른 데이터 조각에 같은 코드를 병렬 실행
        - 조각들이 완전히 독립이면, 이것이 곧 embarrassingly parallel이 된다.

# Mutliprocessing과 MutliThreading

- 프로세스와 스레드
    - 프로세스 : 실행 중인 프로그램의 독립적인 인스턴스
        - 각 프로세스는 자신만의 메모리 공간을 가진다.
    - 스레드 : 하나의 프로세스 안에서 돌아가는 실행 흐름
        - 같은 프로세스 안의 `스레드들은 메모리를 공유`한다.
            - 데이터를 주고 받기 쉬움
            - 같은 변수를 동시에 건드리면, race condition 발생
- MultiThreading과 MultiProcessing
    - MultiProcessing : 프로세스 자체를 여러 개
        - 각 프로세스 메모리 독립, 생성 비용 높음, 통신을 위한 매커니즘 필요
        - 계산이 많은 CPU Bound Process에 유리하다. 
            - CPU-bound 작업은 어차피 기다림이 없기 떄문에, 계산 능력 자체를 늘리는 것이 유리하기 때문
            - 프로세스를 여러 개 띄우면 OS 실제로 동시에 계산을 진행한다.
    - MultiThreading : 한 프로세스 안에 스레드 여러 개
        - 메모리 공유, 생성 비용이 낮음
        - IO bound process에 유리하다.
            - IO bound 작업은 기다림이 병목이기 때문에, 그 시간에 다른 작업을 하게 하는 것만으로도 시간이 크게 줄어든다.
            - 실제로 동시에 하는 것이 아님에도 동시에 하는 것처럼 보일 수 있다.
    - 특히 Python에는 GIL이 존재하기 때문에 한 번에 하나의 스레드만 Python을 실행할 수 있다.
        - CPU-bound에 Multi Threading 써봤자, GIL 때문에 한 번에 하나만 계산하므로 빨라지지 않는다.
            - BUT Multi Processing은 프로세스 별로 GIL이 있기 때문에 CPU-bound에서도 실제로 빨라진다.
        - IO-bound에 Multi Threading을 쓰면, Thread가 IO를 기다리는 동안 GIL을 놓아주기에 빨라진다.
        - 따라서 Python에서는 IO는 스레드, CPU는 프로세스가 공식처럼 사용된다.
### ETL pipeline
> ETL에 Parallel Processing을 어떻게 적용해야 할까?
- Extract 
    - 시간의 대부분을 네트워크, 파일, DB Connection을 기다리는데 사용
        - 즉, **IO Bound task**
    - Concurrency Fit : Multi Threading이 잘 맞음
        - 비동기적으로 한 스레드가 응답을 기다리는 동안 다른 스레드가 다른 소스에 요청을 보내는 식으로 처리
    - Hardware Fit : 네트워크 대역폭, 연결 제한, 메모리 등이 중요
        - CPU 코어는 아무리 늘려봤자 소용이 없다. (CPU Bound가 아니니까)
        - Memory는 왜 중요할까?
            - 소스가 엄청 크다면, 그것을 한 번에 올릴라면 Memory가 커야한다.
- Transform
    - 파싱, 계산, join, 집계 등에 시간 사용
    - Concurrency Fit : Multi Processing이 잘 맞음
        - 동시에 여러 코어를 사용하는 것
    - Hardware Fit : 코어와 메모리를 늘리기
        - 하지만, shuffle이 많이 생기면, IO bound로 변한다.
            - shuffle : 데이터를 키 기준으로 재배치하는 과정
                - ex, groupBy 혹은 join하면 같은 것을 가진 레코드들이 전부 같은 노드에 모임
                    1. 이때 각 노드가 자기 데이터를 분류해서 로컬 디스크에 작성
                    2. 그 데이터를 네트워크 담당을 통해 노드로 전송
                    3. 받는 쪽이 다시 읽어서 처리
                - shuffle의 비용은 disk IO 뿐만 아니라 노드간 네트워크 전송도 있다.
                    - 따라서 셔플 = Disk IO + 네트워크 IO
        - shuffle이 생기면 성능이 많이 떨어지므로, Spark Tuning의 핵심이다.
- Design Principle
    > 워크로드를 모델이나 머신에 맞춰라. Mismatch는 성능을 낮춘다.
    - 실무에서 자주 발생하는 Anti-Pattern
        - CPU-Bound여야 하는 Transform 안에서 Record 마다 외부 API를 호출하는 경우
            - ex, 주식 보고서 작성 
                - 장 중 가격을 환율에 맞춰 변환해야 함. 그럼 이걸 Transform에서 어떠한 기준으로 어떻게 변환해야 할까?
                    - 이때 실시간 환율 API 이런걸 쓰면 모든 가격에 호출해야 하므로 매우 문제가 발생한다.
                - Cache를 둬서 임시로 저장해서 사용할 수도 있고 환율을 특정 값으로 고정할 수도 있고 여러 방법을 고안해야한다.
            - 해결책 4가지
                1. Minimize : 호출 자체를 줄이기
                2. Cache : 같은 입력 결과를 캐싱하여 중복 호출 제거
                3. Batch : 한 건씩 아니라 1000건씩 묶어 호출
                4. Pre-join : 필요한 참조 데이터를 미리 통째로 받아와서 로컬에서 조인
    - 상황을 보고 맞는 머신/API를 선택해야 한다. 실시간 API를 함부로쓰지말자.

# Distributed Computing
- Parallel Computing 과 Distributed Computing의 차이
    - Parallel Computing
        - 단일 머신에서 여러 개의 Processor에 Task를 나누어 수행
    - Distributed Computing
        - 여러 머신을 네트워크로 연결해 하나의 시스템처럼 동작시키는 방식
        - 여러 머신 각자의 Processor로 Task를 나누어 수행
        - 장점 
            1. Scalability & Modular Growth
                - Scale-out이 가능하기 때문에 트래픽이 늘어나면 그냥 머신을 하나씩 더 추가해서 Modular Growth가 가능하다.
                - 처음부터 큰 시스템을 구축할 필요가 없다.
            2. Fault Tolerance & Redundancy
                - 분산 시스템은 여러 노드에 데이터와 기능을 복제해 두기 때문에, 일부 노드가 고장나도 시스템 전체는 계속 동작
- High Availability & Fault Tolerance
    - High Availability : 목표, 시스템이 중단 없이 항상 사용 가능한 상태를 유지하는 것
    - Fault Tolerance : 수단, High Availability를 달성하기 위한 수단, 장애가 발생해도 시스템이 견딜 수 있게 설계하는 기법
        - Redundancy(이중화/중복성) : 장애 허용을 구현하는 가장 기본적인 방법
            - 서버, DB, 네트워크 장비, 전원 등을 여러 개 두는 것
            - ex, DB를 Primary-Replica 구조로 복제해 두면, 주 서버가 죽어도 복제본이 즉시 역할을 이어받을 수 있다.
        - SPOF(Single Point of Failure) 없애기
            - SPOF : 하나가 고장나면 시스템 전체가 멈추는 지점
            - SPOF를 식별하고 제거하는 것이 설계의 핵심 과제 (ex, 로드밸런서 이중화)
        - Disaster Recovery(재해복구)
            - 화재, 지진, 정전 등의 재해 상황에서 시스템과 데이터를 복구하는 계획과 체계
                - 보통 지리적으로 떨어진 다른 Region에 백업이나 대기 시스템을 두어서 구현
            - 지표 : RTO(Recovery Time Objective), RPO(Recovery Point Objective)
# Docker

- VM과 차이
    - Container는 각 Application마다 따로 OS가 들어가지 않는다. (Host의 kernel을 공유)

    2. 직접 하나씩 작성하며 잘 작동하는지 계속 확인
    3. 완벽한 Product를 제출 (시행착오 제출 XXX)
- Docker는 과제 하면서 따로 학습할 것이라 그냥 수업에 나온 것만 간단히 정리했다.

# Data Product
### 정의
- Reusable data asset : 여러 팀과 여러 용도에서 반복적으로 사용되는 자산.
- Bundles data together with everything needed : 데이터와 그것을 쓰는데 필요한 모든 것을 함께 묶어 전달 
    - 데이터 프로덕트와 데이터셋의 결정적 차이점
    - 단순 테이블이나 파일만 주는 것이 아니라 다음을 함께 제공
        1. Metadata (이 데이터가 무엇인지, 언제 갱신되는지, 출처가 어디인지)
        2. schema와 문서 (컬럼의 의미, 데이터 타입)
        3. 접근 방법 (API, SQL 인터페이스)
        4. 품질 보증 (데이터 품질 기준, SLA)
        5. 접근 제어 (누가 쓸 수 있는지)
- Independently usable by authorized consumers : 권한있는 consumer가 독립적으로 이용가능
    - 스스로 찾아서 이해하고 사용할 수 있게 만들어야 한다는 의미

### 전통적인 프로젝트 중심과의 차이
1. Project-Driven
    > 비즈니스 Use-Case마다 파이프라인 전체를 새로 만든다.
    - 문제점
        1. Use-Case마다 자신만의 사일로를 가짐
            - 중복 작업 발생
        2. 작업간 일관성 없을 수 있음
        3. 확장 비용 : Use-Case마다 선형으로 증가

2. Product-Driven
    > 여러 Use-Case가 같은 데이터 프로덕트를 사용한다.

# W2M1
## multiprocessing
> 프로세스 기반 병렬 처리를 제공하는 파이썬 표준 라이브러리 (스레드가 아닌 별도 프로세스 생성)
- 장점
    - Python은 프로세스 단위로 GIL을 관리하므로, GIL에 병렬 처리가 막히지 않는다.
        - GIL : 하나의 프로세스 안에서 한 시점에 단 하나의 스레드만 Python 바이트 코드를 실행하도록 강제하는 것
- 단점
    - Process끼리는 메모리를 공유하지 않기에, 데이터를 주고받으려면 직렬화(pickle) 이후, IPC로 전달해야 하고, 프로세스 생성비용이 Thread보다 크다.
- 선택 기준
    - CPU-bound : multiprocessing
    - I/O-bound : threading / asyncio

### Multiprocessing의 Pool 개념
> 워커 프로세스들을 미리 정해진 개수만큼 만들어 놓고, 작업 목록을 분배하는 관리자
- 작업마다 프로세스를 새로 만들고 죽이는 대신, 고정된 워커 풀을 재사용
- 주요 메서드
    1. `map(func,iterable)` : blocking 방식
        - 모든 작업이 끝날 때까지 기다렸다가 결과를 입력 순서대로 반환
        - 인자가 하나인 함수만 가능하다.
    2. `map_async(func,iterable)` : non-blocking 방식
        - 즉시 AsyncResult 객체를 반환하고, 나중에 `.get()`으로 결과 받음
    3. `apply(func,args)`/`apply_async(func,args)` : 작업 하나를 워커에 넘김
        - 여러 인자를 받는 함수에 사용 가능
        - apply는 병렬이라 효과가 없기에 실무에서는 apply_async를 사용한다.
    4. `starmap(func,iterable)` : map과 같지만, 각 항목을 언패킹해서 넘긴다.
    5. `imap(func,iterable)` : map의 lazy 버전
        - 결과를 한 번에 보여주지 않고, 완료되는대로 iterator로 하나씩 보여줌
        - 메모리 효율이 좋다.
- 주의 사항
    1. if __name__ == "__main__" 사용
        - window와 macos는 spawn으로 시작하기 때문에 자식 프로세스가 메인 모듈을 다시 import하여 무한 생성한다.
    2. 전달되는 함수와 인자는 pickle 가능해야 한다.
        - pickle이 안 되는 것(람다, 중첩 함수, 열린 파일 핸들 등)은 워커에 넘길 수 없다.
    3. Context Manager 사용을 권장한다.
        - 사용하지 않으면 close -> join을 해줘야 하므로, context manager를 사용하자.
    4. print 출력 순서 보장 X
        - 당연히 여러 프로세스가 동시에 실행되므로, 순서가 달라지거나 섞일 수 있다. 걱정하지 않기
## W2M2
### Process
> OS 프로세스 하나를 표현하는 객체
- Pool과 Process의 차이
    - Pool : Worker 여러 개를 묶어 작업을 자동으로 분배하는 고수준의 관리자
    - Process : Pool 아래의 기본 단위, 프로세스 하나의 생성, 시작, 종료를 직접 제어하는 용도
        - Pool도 Process들을 만들어서 관리하는 구조로 되어있다.
        - Interface가 Thread와 동일하게 설계되어 있어서 스레드를 사용해본 적이 있다면, 사용 가능하다.
- Process 생성
    ```python
    from multiprocessing import Process

    def worker(x, y, mode="fast"):
        print(x+y, mode)
    
    p = Process(target=worker, args=(1,2), kwargs={"mode": "slow"}, name="worker-1")
    ```
    - target : 새 프로세스에서 실행될 함수 (호출 가능 객체)
    - args : 위치 인자 튜플, 인자가 하나여도 튜플 형태 유지해야 한다. (a,) 이런식으로
        - 여기에 작성된 인자가 target에 들어간다. 즉, args를 안 적으면 함수에 아무런 인자가 전달되지 않고, 함수는 기본 값으로 동작하게 된다.
    - name : 프로세스 이름(디버깅용, 미지정시 자동 부여)
    - daemon : 데몬 여부
        - 데몬으로 지정(daemon=True)하면, 메인 프로세스가 종료될 때 해당 프로세스도 자동으로 강제 종료된다.
            - 백그라운드 모니터링과 같이 메인이 없어질 때 같이 사라져도 되는 작업에 설정한다.
        - 기본값인 경우, main이 join하지 않아도 종료시점에 기다려준다.
            - 그래도 명시적 join을 사용하는 이유는 순서를 코드에서 보장하기 위해서 + 데몬 프로세스는 기다려주지 않기 때문.

- Process 생명주기
    > 프로세스의 생명주기를 제어하는 메서드
    - start() : 실제로 프로세스를 생성하고 target 함수 실행을 시작하는 메서드
        - 한 객체당 단 한 번만 호출 가능 (두 번 호출시 예외)
        - p.start()와 p.run()은 다르다.
            - p.run()은 현재 프로세스에서 함수가 그냥 실행되는거, 병렬 처리가 안됨
    - join(timeout=None) : 해당 프로세스가 끝날 떄까지 현재(호출한) 프로세스를 블록한다.
        - 모든 프로세스가 끝난 뒤 스크립트가 종료되도록 보장
        - timeout을 주면, 최대 해당 시간만 기다리고, 타임아웃이 지나도 프로세스를 죽이지 않고, 기다리지만 않는다.
    - is_alive() : 프로세스가 아직 실행 중인지 bool 반환
    - terminate() / kill() : 강제 종료
        - terminate: `SIGTERM` / kill: `SIGKILL` 을 보낸다.
        - 강제 종료된 프로세스 : `finally 블록`이나 `자원 정리 코드를 실행하지 못함.`
            - Queue나 Lock을 가진 채로 죽어 `교착 상태가 발생`할 수 있다. -> 신중하게 사용
    - close() : 프로세스 객체의 자원을 해제, `종료된 프로세스`에만 `호출 가능`

- Process 병렬 실행
    - 잘못된 방식 : 하나 실행 - 종료를 반복
        ```python
        for item in data:
            p = Process(target=worker, args=(item,))
            p.start()
            p.join() # 이렇게 하면, 이 join이 끝나야 다음 Process가 실행된다.
        ```
    - 잘 된 방식 : 한 번에 다 켜고, 몰아서 join
        ```python
        processes = []
        for item in data:
            p = Process(target=worker, args=(item,))
            p.start()
            processes.append(p)
        for process in processes:
            process.join()
        ```

- Process간 데이터 주고 받기
    - Process는 메모리를 공유하지 않으므로, 자식 Process의 결과를 부모가 받으려면 별도 수단이 필요하다.
        - target 함수의 return 값은 그냥 버려진다.
    - 방식
        1. Queue : 자식이 삽입, 부모가 추출 / 여러 프로세스가 동시에 해도 안전
        2. Pipe : 양방향 1:1 통신 / Queue보단 빠르지만 두 끝점만 지원
        3. Value/Array : 공유 메모리에 놓인 단일 값/배열, 숫자 카운터 등 단순한 공유 상태용, 동시 수정 시엔 내장락으로 보호해야 한다.
        4. Manager : 공유 list, dict 등 고수준 공유 객체를 제공하는 방식 / 편하지만 별도 서버 프로세스를 거치므로 느리다.
    - 동기화 도구
        - Lock, Semaphore, Event, Barrier 등 threading과 같은 인터페이스로 제공한다. Lock을 이용하여 직렬화가 가능하다.

- 서브클래싱
    > target 함수를 넘기는 대신, Process를 상속해서 run 메서드를 오버라이드 하는 스타일
    ```python
    class MyWorker(Process):
        def __init__(self, data):
            super().__init__()
            self.data = data
        def run(self): # start()가 호출되면 이 run()함수가 실행된다.
            print(f"processing {self.data}")
    w = MyWorker([1,2,3])
    w.start()
    w.join()
    ```
    - 상태를 갖는 복잡한 워커를 만들어야 할 때 유용

- 시작 방식 
    > 프로세스가 어떻게 만들어지는지 차이 이해
    1. spawn : 완전히 새 Python 인터프리터를 띄우고 필요한 것만 pickle로 전달
        - 느리지만, 안전한 방식
        - Window/MacOS의 기본 값(Python 3.14부터는 Linux도 spawn계열)
    2. fork : 부모 프로세스를 통째로 복제
        - 빠르고 상태를 그대로 물려받음
        - 멀티스레드 환경에서 복제하면 lock이 잠긴채로 복제되는 위험이 있음
        - Python 3.13 이전의 Linux 기본값
    3. forkserver : 꺠끗한 서버 프로세스를 하나 띄워두고 거기서 fork
