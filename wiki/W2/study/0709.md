### 모델링과 도메인

<details>
<summary>데이터 중심 모델링</summary>

> `이 시스템이 다루는 데이터는 무엇인가?`를 답변함

- 먼저 정의 : 엔티티 / 속성 / 관계
- 1차 산물 : 스키마 
- 규칙 원본 : DB에 적음
    - application code가 어떻게 변하든, DB에 저장된 데이터와 제약 조건이 규칙
- 배경 : Enterprise IT에서 하나의 DB를 여러 app이 동시에 접근하게 되었음
    - 추가로 앱은 계속 추가/변경되기에 `데이터를 잘 설계하는 것이 중요하다.`로 결론

</details>

<details>
<summary>도메인 중심 모델링</summary>

> `이 시스템은 어떤 업무 행동과 규칙을 수행하는가?`를 답변함
- 먼저 정의 : 행동과 규칙
- 테이블은 그냥 도메인 객체를 저장하기 위한 수단
    - 도메인 모델과 테이블이 1:1이 아닐 수 있음
        - ORM이 객체를 매핑한 부산물이기 때문
    - CQRS처럼 쓰기 모델과 조회 스키마를 아예 분리하기도 한다.
        - 하나의 데이터 모델이라는 전제가 깨짐
- 규칙 원본 : 도메인 객체 코드
- 배경
    - 복잡한 업무 로직의 유지보수
        - 데이터 중심일 땐 업무 규칙이 Service계층 여기저기에 있어서 유지보수가 어려웠음
    - OOP 원칙(데이터와 데이터를 다루는 행동을 한 객체에 묶는다.)을 업무 로직에 적용하는 것
</details>


<details>
<summary>데이터 중심 모델링과 도메인 중심 모델링의 차이</summary>

> `규칙 원본이 어디에 있는가`

- 데이터 중심일 땐
    - DB 제약 조건
    - 클래스는 테이블의 복사판(DTO)
    - 스키마 변경은 신중하게 합의 후 진행
- 도메인 중심일 땐
    - 도메인 객체의 메서드
    - 행동을 가진 살아있는 모델
    - 도메인 진화에 따라 유연하게 진행
</details>

<details>
<summary>데이터 중심 모델링과 도메인 중심 모델링 판단 기준</summary>

> 데이터를 얼마나 공유하는가? -> 데이터 중심 / 업무 규칙이 얼마나 복잡하고 자주 바뀌는가? -> 도메인 중심

- 데이터 중심 : 여러 App이 공유하는 자산을 최적화
    - 데이터가 앱보다 오래 사는 경우
    - 여러 부서가 앱을 공유하는 경우
- 도메인 중심 : 한 App 안의 업무 로직 자산을 최적화
    - 하나의 서비스가 복잡한 업무 규칙을 갖는 경우
    - 요구 사항이 자주 바뀌는 경우
    - 여러 화면/API가 같은 규칙을 공유해야 하는 경우

- 최근
    - MSA로 변해가며, 서비스 각각이 자신의 DB를 갖는 방향으로 변해간다.
        - 여러 앱이 DB를 공유하는 상황이 사라짐
            - 데이터 중심 패러다임이 약해지고, 도메인 중심 패러다임이 자연스러워진다.
    - 반대로, 데이터 레이크나 웨어하우스처럼 분석을 위해 모으는 경우 다시 데이터 중심 모델링이 중요해진다.
</details>

<details>
<summary>빈약한 도메인 모델이란?</summary>

> 테이블 구조를 그대로 클래스로 옮겼을 때, 발생하는 `속성만 있고 행동이 없는 도메인 객체`

- 테이블 구조를 도메인으로 옮기면, 속성과 getter/setter만 존재
    - 로직이 하나도 없고, 다 Service 레이어에서 구현함
        - Service Layer에서 구현하니까 규칙들이 흩어지고 중복된다는 문제가 존재한다.
            - 중복은 항상 인간의 실수를 유발함. (정규화 갱신 이상 참고)
</details>
<details>
<summary>빈약한 도메인 모델의 대안</summary>

> 행동(규칙)을 데이터와 함께 두기

- setter가 없이, 상태를 바꾸는 것은 메서드로만 가능하게 함
- Service 레이어가 얇아질 수 있음.
</details>

<details>
<summary>정수 타입 종류 및 표현 범위, 사용 예시를 말하라.</summary>

> 정수는 `표현 범위`로 고른다. 무조건 INT를 사용하는 것은 낭비, 위험할 수도

1. TINYINT (1byte)
    - ex, 나이, 상태 코드
2. SMALLINT (2byte)
    - ex, 재고 수량
3. INT (4byte)
    - ex, 대부분의 ID
4. BIGINT (8byte)
    - ex, 대용량 PK, 조회수

> 값의 범위를 넘을 일이 없다면 큰 dtype을 사용할 필요가 없다.
</details>


<details>
<summary>소수 타입 종류 및 특징, 사용 상황</summary>

> 금액이나 수량처럼 정확히 맞아야 하는 것은 `DECIMAL`을 사용하라, `FLOAT/DOUBLE`은 근사치.

- DECIMAL
    - 십진수 자릿수를 그대로 보존하는 형식
    - CPU에서 계산하는게 아니라 DBMS로 구현한거
        - 사람이 연산하듯, 자릿수를 쪼개서 여러 단계로 나누어 정수 연산 처리
    - 이러한 이유로 많은 연산이 이루어질 때는 속도가 많이 차이난다.
    - 저장공간 또한, 더 많이 차지할 수 있다.
        - 정수부와 소수부를 분리하여 9자리씩 묶어서 9자리당 4바이트씩 차지
            - 나머지 자릿수는 더 작은 바이트 사용
        - 대신 정밀도(p)를 작게하면, DOUBLE보다 작아진다.
            - p가 18보다 커지면 DOUBLE보다 커짐
- FLOAT/DOUBLE
    - IEEE 754에 따른 국제 표준 이진 형식으로 소수점 저장
        - 지수/가수
    - CPU로 직접 연산
        - DECIMAL보다 연산 속도가 매우 빠름
        - 근사치도 괜찮다면 FLOAT/DOUBLE이 낫다.
</details>


<details>
<summary>CHAR와 VARCHAR의 차이가 무엇인가?</summary>

> 저장 방식이 달라서 저장하는 속성에 맞게 설정해야 한다.

- CHAR(N) 
    - 고정길이 : 모자라면 페이지 내에서 차지하고 있는 공간 중 빈 공간을 공백으로 놔둠
    - 속도 : 고정 길이라 약간 빠름
    - 적합한 데이터 : 항상 길이가 같은 값 / 길이는 다르지만 갱신이 잦은 값
        - 고정 길이로 위치가 변경될 일이 없음
    - 예시
        - 주민번호/우편번호/상태코드
- VARCHAR(N)
    - 가변길이 : 실제 길이만큼만 저장
        - 실제 길이를 나타내는 길이 정보(1-2바이트)를 앞에 붙임
    - 속도 : 상대적으로 느릴 수 있음
    - 적합한 데이터 : 길이가 들쭉날쭉하고, 갱신이 거의 없는 값
    - 예시
        - 이름/주소/이메일
</details>

<details>
<summary>VARCHAR에 대해 설명해주세요.</summary>

> VARCHAR는 가변 길이 배열로, 저장된 길이만큼만 저장된다. 만약 현재 길이보다 더 긴 길이로 갱신하는 경우에는 페이지 위치 변경이 이루어진다.

- DBMS마다 구현은 다르지만, Oracle의 경우 이동하기 직전 위치에 포인터를 남겨서 이동되었음을 표시하고, MySQL InnoDB의 경우에는 페이지 분할 방식으로 페이지를 둘로 쪼개서 행들을 나눠 담는다.
    - MySQL 추가 : InnoDB는 행은 반드시 PK 순서대로 연속된 페이지에 있어야 한다는 규칙이 있기 때문에 포인터를 다른 곳으로 옮기지 못한다.
        1. 페이지를 둘로 나눔
        2. 새 페이지로 현재 페이지의 절반을 옮김
        3. 기존 페이지에 여유가 생겼으니 값을 갱신
        - 이렇게하면 빈 페이지가 증가하면서 범위 조회 I/O가 증가한다.
            - 곧 속도 저하로 이어짐
    - 각각 속도와 분할비용/페이지 공간 활용률/속도저하에서 비용이 발생한다.
</details>

<details>
<summary>VARCHAR와 TEXT를 각각 언제 사용해야 하나요.</summary>

> 인덱스 관련 제약의 차이로 WHERE나 ORDER BY처럼 인덱스를 활용해야 하는 경우 VARCHAR를 사용합니다.
>> TEXT/CLOB은 길이를 가늠하기 어려운 본문 같은 경우에 사용합니다. VARCHAR는 길이의 상한을 명확히 알고, 길이가 들쭉날쭉하며 갱신이 거의되지 않는 경우 사용합니다.

- VARCHAR는 인덱스를 걸기 쉽고, 정렬 비교가 빠르다는 장점
- TEXT는 인덱스에 제약이 있고, 조회 성능도 떨어진다.
</details>

<details>
<summary>날짜 타입 종류 및 사용 예시 설명</summary>

> 시간까지 필요한지에 따라 사용이 다름
>> 필요 : DATETIME / 필요 없음 : DATE
>>> 괜히 DATETIME 남발 시, 비교가 지저분해지고 힘들어짐

- DATE(3byte) : 연/월/일
    - 표현 범위 : 1000-01-01 ~ 9999-12-31
- YEAR(1byte) : 연
    - 표현 범위 : 1901 ~ 2144
- TIME(3byte) : 시:분:초
    - 표현 범위 : -838:59:59 ~ 838:59:59
- DATETIME(5byte) : 날짜 + 시간
    - 표현 범위 : 1000-01-01 ~ 9999-12-31
- TIMESTAMP(4byte) : 날짜 + 시간 (타임존 변환 동작)
    - 표현 범위 : 1970-01-01 ~ 2038-01-19 (UTC)
    - 날짜를 자릿수로 저장하지 않고, UTC부터 흐른 second를 저장하기 때문 
        - 4바이트 정수 한계로 2038저 날짜를 넘지 못한다.
    - 전 세계 사용자를 다루는 서비스라면, 타임존 처리 방식을 DBMS 문서에서 확인하자
</details>

- BOOLEAN 
    - 장점 : 의미가 명확하다.
    - 단점 : Oracle 등 일부 DBMS에서 지원 X
- Char(1)
    - 장점 : 모든 DBMS에서 동작, 리버스 시 의미 파악이 쉽다.
    - 단점 : 값 검증을 CHECK 제약으로 추가해야 함.
        ```sql
        is_deleted CHAR(1) DEFAULT 'N' CHECK (is_deleted IN('Y','N'));
        ```
- 코드 테이블 참조
    - 장점 : 상태가 2개 이상이 되어도 대비 가능
    - 단점 : 조인 설계 필요 / 설계 비용 증가
</details>

# W2M5 논문 소감

> 훈련 데이터셋을 어떤 기준으로 어떻게 수집했으며, 전처리와 정제 과정을 어떻게 수행하여 좋은 결과를 얻을 수 있었는지 궁금하다. 잘 정제되고 전처리된 소셜 미디어 데이터가 기존 전문가들의 예측 방식보다 더 나은 결과를 보였다는 점이 흥미롭다. 내 생각에 이러한 결과가 가능했던 이유는, 출구 조사에 응하는 사람은 전체 유권자 중 일부에 불과하고 응답자 집단이 편향적일 가능성이 있는 반면, 소셜 미디어 데이터는 표본의 규모가 훨씬 크기 때문인 것 같다. 이러한 접근법은 사람들이 예측을 수행하는 다양한 분야에 적용할 수 있을 것이다. 예를 들어, 회사가 광고 모델을 선정할 때 광고 모델의 브랜드 가치(대중 선호도)를 예측하는 데에도 활용할 수 있을 것 같다.

# 수업

- try-exception
    ```python
    while True : # 무한 수행
        try: # 수행 시도
        except: # 예외 발생했네
            break
        else: # 잘됐으니까 마무리
    return True
    ```
    - try 안에 최소한의 코드가 있어야 한다.
        - 진짜 잡고 싶은 한 개
    - 잘쓴 코드를 많이 읽고, 리뷰하자
- 동시에 돌리는 것의 문제를 항상 인식하기
    - atomic하게 동작해야 한다.

- 웹 스크래핑을 해봐야 하는 이유
    - 필요한 데이터가 있는데, 못 구하는 경우가 생각보다 많고, 이때는 무조건 웹 스크래핑을 해야한다.
        - 사업적 가치를 증명하기 위해서는 프로토타이핑을 위해 필요하다.
        - 이러한 경우를 대비하기 위함이다.
    - 손이 빨라야 한다.
        - web scraping / EDA
    - 다양한 형식의 데이터를 다뤄야 하는 경우가 너ㅓ무 많다.
        - 세상은 아름답지 않다.

- 데이터 프로덕트의 대표적 예시
    - 유튜브의 데이터 API
        - 평균 시청 지속시간, 이탈률 등

- 고객의 선택 기준
    - 하이즌 팩터 / 모티베이션
        - 하이즌 팩터 : 일정 수준 이하로 내려가면 선택 못하게 되는 것
            - 이건 특정 기준 아래로 가면, 선택을 안하지만
            - 특정 기준을 월등히 넘는다해도 선택의 이유가 되지는 않는다.
        - 모티베이션 : 살 이유
