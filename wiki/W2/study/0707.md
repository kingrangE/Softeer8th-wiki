### 3. 목록에서 중요 개념을 골라 공식 문서 + Claude를 병행하며 정리
1. Queue란(multiprocessing)? 
    - IPC를 위한 FIFO채널
        - 프로세스는 각자 독립된 메모리 공간을 가지므로 변수 공유가 불가
            - 데이터 전달 어떻게?
                - kernel이 제공하는 통로 이용 -> Queue가 이 통로 추상화한 것
    - OS pipe + lock/semaphore로 구현
        - 여러 프로세스가 동시에 put/get 해도 안전하다. (thread 안전+process 안전)
            - thread-safe
                - 여러 스레드가 같은 프로세스 내에서 메모리를 공유하면서 동시에 접근해도 데이터가 깨지지 않는 성질
                - mutex, lock, semaphore, atomic 연산 등을 이용
            - process 안전
                - 여러 프로세스가 동시에 같은 자원에 접근해도 문제가 없는 성질
                - file lock, DB트랜잭션, named semaphore 등으로 동기화  
                    - named semaphore : 커널이 이름으로 식별하는 세마포어
                        - 프로세스는 서로 정보 공유가 안 되므로 named semaphore를 이용
    - Queue에 넣는 모든 객체는 pickle로 직렬화되어 전송 
        - 성능/특성/예외/제약 모두 여기서 파생
    - Queue는 Pipe를 기반으로 Lock과 Buffer를 추가해 만든 것
2. Queue 내부 동작
    - put()
        1. 객체가 내부 버퍼(deque)에 들어가고, put함수는 즉시 return
        2. 백그라운드의 feeder thread가 객체를 꺼냄
            - feeder thread : pipe에 데이터를 넣어주는 스레드라는 의미
        3. pickle.dumps()로 직렬화
        4. 직렬화된 데이터를 pipe로 삽입
    - get()
        1. Pipe에서 데이터를 읽음
        2. pickle.loads()로 역직렬화하여 객체 복원
    - 주의점
        > 위의 내부 동작 과정으로 인해 생기는 주의점
        1. put의 반환이 전송 완료를 의미하는게 아니다.
            - 직렬화와 파이프 기록은 비동기
            - put 직후에 상대측에서 get_nowait()하거나 empty()를 호출하면 Empty로 나올 수 있다.
                - 공식 문서에서도 지연이 있을 수 있다고 써있음
        2. pickle 에러가 잡기 어려운 시점에 터짐
            - 위에서 보면, put 함수 반환 이후에 pickle 직렬화를 한다.
            - pickle 직렬화가 불가능한 객체를 넣으면, 에러가 발생하는데 이게 put함수 반환 이후이기에 try/except로 잡을 수 없다.
        3. 직렬화 속도가 병목
            - 큰 크기의 객체를 직렬화하면, 직렬화/역직렬화와 파이프 복사에 걸리는 시간이 병목이 된다.
        4. Pipe Buffer 크기 제한
            - maxsize를 무제한으로 설정해도, Pipe 버퍼가 가득차면, feeder thread가 막힌다. 
3. Queue 예외
    1. queue.Empty
        - 발생 조건 : 빈 큐에 get
    2. queue.Full
        - 발생 조건 : 가득찬 큐에 put
    3. ValueError
        - 발생 조건 : close된 큐에 get/put
    4. OSError/EOFError/BrokenPipeError
        - 발생 조건 : 상대 프로세스가 죽어 파이프가 끊긴 경우에 닫힌 커넥션을 사용할 때
    5. pickle 관련 (PicklingError, TypeError 등)
        - 발생 조건 : 직렬화 불가능 객체 put (feeder thread에서 발생)
    - 주의
        1. Empty와 Full은 queue 모듈 정의 
            - multiprocessing이 자체 예외를 만들지 않고 재사용하기 때문 
            - `from queue import Empty, Full` 사용하기
        2. Empty 발생과 논리적으로 큐가 빔은 다름
            - feeder thread 지연으로 아직 도착 안 한 것일수도 있음
            - Empty가 한 번 났으니 작업 끝이라는 판단 XXX
                - 종료 판단은 sentinel
4. 자주 사용하는 메서드
    ```python
    # 삽입 함수, 가득차면 Block(대기), Timeout시 Full
    put(obj, block=True, timeout=None)
    # 삽입 함수, 가득차면 바로 Full (put(obj,block=False)와 같음) 
    put_nowait(obj)

    # 추출 함수, 비면 Block(대기), Timeout시 Empty
    get(obj, block=True, timeout=None)
    # 추출 함수, 비면 바로 Empty (get(obj,block=False)와 같음)
    get_nowait(obj)

    # 큐 크기
    qsize()

    # 상태 (100% 신뢰는 하면 안된다.(지연 존재))
    empty()
    full()

    # 이 프로세스가 더 이상 put하지 않음을 선언. 버퍼의 잔여 데이터를 파이프로 flush
    close()

    # feeder thread 종료(flush 완료)까지 대기 (close() 이후에 호출 가능)
    join_thread()

    # 프로세스 종료 시 flush를 기다리지 않게 함. (버퍼의 데이터 유실 가능. 데드락 회피용 최후 수단)
    cancel_join_thread()
    ```
5. Simple Queue
    - Queue를 극단적으로 단순화한 버전
        - 제공 메서드 : get/put/empty/close
    - Queue와 다르게 내부 버퍼와 feeder thread가 없다.
        - put 함수가 lock을 잡고 직접 pickle에 쓴 후, 파이프에 삽입함.
        - 따라서, put 함수 결과가 파이프 삽입되었음을 의미 (비동기 문제 해결)
        - But, 문제
            - Pipe Buffer가 가득차면, put이 Block되고, timeout으로 빠져나오지 못한다.
    - 언제 사용?
        - Simple Queue : 자식 Process가 결과 1,2개 반환해주는 단순, 저빈도 통신일 때 
        - Queue : 흐름제어가 필요한 지속적 Stream일 때
6. Joinable Queue
    - Queue의 서브 클래스로, task_done()과 join()메서드 추가
    - 미완료 작업 카운터(내부적으로 세마포어와 조건 변수)를 유지하기 때문에 처리 여부까지 검사 가능하다. 
        - 일반 Queue는 데이터가 빠졌는지만 알 수 있고, 처리까지 끝났는지는 알 수 없다. 
        - put : 미완료 카운트 + 1
        - get : 미완료 카운트 - 1
        - 미완료 카운트가 0이 되면, join()에서 대기하던 프로세스가 깨어남
    - 규칙
        1. get 1회당 task_done 1회 - 처리 중 예외 발생해도 done되게 finally에 작성
        2. get 횟수보다 많이 호출하면 Error(ValueError: task_done() called too many times)        
        3. 호출 안하면 join()이 영원히 블록된다.
    - 예시 코드
        ```python
        import multiprocessing as mp

        def worker(q):
            while True:
                item = q.get()
                try:
                    process(item)
                finally:
                    q.task_done()

        if __name__ == "__main__":
            q = mp.JoinableQueue()
            for i in range(100):
                q.put(i)

            for _ in range(4):
                mp.Process(target=worker, args=(q,), daemon=True).start()

            q.join()   # 100건 전부 task_done 될 때까지 블록
            # daemon 프로세스이므로 메인이 끝나면 워커도 종료됨
        ```
7. Queue를 사용하는 상황
    1. 생산자-소비자 / 워커 풀 : 생산 프로세스가 작업을 넣고 워커 N개가 꺼내서 처리 
        - 놀고 있는 워커가 먼저 가져가므로 로드 밸런싱으로 동작
    2. Process의 반환값 수집 : 프로세스 결과를 Queue에 put하고 부모가 get
        - 그냥 mp.Process는 target 함수의 return 값을 부모에게 전달할 수 없음
    3. 단계형 파이프라인 구현 : 수집 -> 처리 -> 저장 프로세스를 큐 2개로 연결하여 사용
        - 각 단계가 독립적으로 병렬 동작
    4. backpressure : maxsize를 지정하면 소비가 느릴 때 생산자가 자동으로 블록되어 메모리 폭주 방지 가능
    5. 멀티프로세스 로깅 집약 : 각 프로세스가 로그 레코드를 mp.Queue에 넣고, 단일 리스너 프로세스가 꺼내 파일에 쓴다
        - 여러 프로세스가 같은 파일에 직접 쓸 때 로그가 뒤섞이는 것을 방지
        - logging 모듈의 QueueHandler/QueueListener가 이 큐를 받음
8. Sentinel 패턴 (종료 신호)
    - 아까 위에서 q.empty() 대신 종료 판단은 sentinel로 해야 한다고 말했었다.
    - 아래와 같이 queue에서 worker가 SENTINEL(None)을 꺼내면 종료되도록 하자. (iter(callable, sentinel)은 callable에서 sentinel값을 가져오면 종료되는 구조)
        ```python
        SENTINEL = None

        def worker(q):
            for item in iter(q.get, SENTINEL):   # SENTINEL을 받으면 루프 종료
                process(item)

        for _ in range(num_workers):
            q.put(SENTINEL)
        ```
9. Deadlock
    - 공식문서에서 아래처럼 이용하면 데드락이 발생한다고 써있다.
        ```python
        def f(q):
            q.put(large_data)

        if __name__ == "__main__":
            q = mp.Queue()
            p = mp.Process(target=f, args=(q,))
            p.start()
            p.join()        # 데드락 가능
            obj = q.get()
        ```
        - 데이터를 put한 프로세스를 Join하기 전에 내부에 있는 q를 get으로 모두 소비해줘야 한다.
            - 즉, get을 join 위에 써야 한다.
10. Queue를 인자로 전달하고 싶다면
    - Queue 객체 자체는 pickle 불가다. 특수하게 Process에 인자로 넘기는 것은 상속이라 특별취급으로 가능하지만, Pool.map이나 apply_async에 인자로 넘기면 RuntimeError가 발생한다.
    - 이를 해결하는게 `mp.Manager().Queue()`
        - 별도의 매니저 서버 프로세스가 실제 큐를 보유하고, 전달하는건 프록시 객체를 전달하는 것
        - 모든 연산이 Manager와 왕복이므로, Queue보다는 느리다.
    - 즉, 
        - Process 직접 관리 : mp.Queue()
        - Pool 워커와 공유 : Manager().Queue()
11. Pipe와 비교
    > Queue가 항상 정답은 아니다.
    - Pipe  
        - 연결 형태 : 두 끝 점간 1:1, 양방향 
        - 안전성 : 같은 끝점을 두 프로세스가 동시에 쓰면 데이터가 꺠질 수 있다. (lock이 없다.)
        - 성능 : 빠르다. (락, 버퍼, feeder 등이 없음)
        - API : send()/recv()/poll()
    - Queue
        - 연결 형태 : 다수 생산자, 다수 소비자
        - 안전성 : lock 내장으로 안전
        - 성능 : 상대적으로 무겁다.
        - API : put/get 계열
12. Queue가 부적합한 경우와 대안
    1. 대용량 데이터 고빈도 전송 : pickle + 파이프 복사 비용이 병목
    2. 단순 스칼라/배열 공유 : mp.Value / mp.Array
    3. 복잡한 공유 자료구조 : Manager().dict() / Manager().list()
    - Queue는 소유권을 넘기는 메시지 전달에 적절
    - 같은 데이터를 함께 보는 상태 공유 경우에는 위의 대안이 적절하다.
14. 버퍼링으로 인한 순서 보장 범위
    - 한 프로세스가 넣은 항목들 사이의 순서는 보장되지만, 서로 다른 프로세스가 동시에 put한 항목들 간의 전역 순서는 보장되지 않는다.
15. 자식 프로세스 비정상 종료시
    - 파이프에 쓰다 죽으면 큐에 남은 데이터가 손상될 수 있고, 이후 get에서 예외가 나거나 영원히 블록
    - 공식 문서도 프로세스를 terminate할 때 큐 손상 가능성을 경고
16. 가비지 컬렉션
    - 큐에 남은 항목이 있는 채로 모든 참조가 사라져도, feeder thread 때문에 프로세스가 즉시 종료되지 못할 수 있다.
    - 명시적으로 비우거나 cancel_join_thread()가 필요

1. WordCloud란?
    > Text Data에서 단어의 출현 빈도를 글자 크기에 비례시켜 2차원 Canvas에 배치하는 시각화 기법
    - 설치
        ```python
        pip install wordcloud
        ```
        - 의존성 : numpy, pillow, matplotlib
    - 용도
        - EDA 초기
        - 발표 자료 분석시, 어떤 용어가 많이 나오는지
    - 부적합한 경우
        - 정확한 빈도 수 비교
        - 순위 판단 필요
2. 내부 동작 원리
    > generate 메서드 호출 시 아래의 3단계 실행
    1. 토큰화 & 빈도 계산
        - 정규식으로 단어 분리
        - STOPWORD 제거, 숫자 제거, min_word_length 미만 단어 제거
        - `normalize_plurals=True`라면, 영어 단수형 기준으로 합산
        - `collocations=True`라면, 자주 붙어 나오는 bigram을 하나로 묶음
    2. 폰트 크기 결정 및 배치
        - 폰트 크기 결정
            1. 빈도를 최대값으로 정규화
            2. relative_scaling에 따라 크기 보간 (빈도 비례 <-> 순위 기반)
            3. 가장 큰 단어부터 시작해서, 안 들어가면 font-step만큼 폰트 크기를 줄여가며 시도한다.
        - 배치 (greedy + integral image)
            - 적분 이미지로 전체 캔버스의 빈자리를 세어 균등 랜덤 선택 진행
            - 충돌 검사를 빠르게 하기 위해 integral image 자료 구조 사용
                - C로 구현
            - 자리 없으면 prefer_horizontal에 따라 90도 회전
    > 배치는 greedy로 확률적으로 진행함 -> 실행마다 결과가 달라짐 -> 항상 일관된 결과 받으려면 **random_state** 설정이 필수다.

3. WordCloud 주요 파라미터
    1. Canvas/Rendering
        - width/height : 캔버스 크기
            - matplotlib fig사이즈랑 별개임
            - figsize를 키우는건 그냥 이미지 늘리기, 해상도를 높이고 싶다면 width랑 height를 올려줘야 한다.
        - scale : 최종 이미지 확대
            - 계산은 width/height로 하고 최종 이미지를 배율 확대
        - background_color : 배경 색
        - mode : RGB가 기본이고, RGBA로 바꾸면 알파채널 지원
        - margin : 단어 사이 여백
    2. Font/Size
        - font_path : 폰트 파일 경로
        - max_font_size : 최대 글씨 크기
            - 기본값이면 캔버스 높이 기준으로 자동 설정
        - min_font_size : 최소 글씨 크기
            - 기본값 4
            - min size로 안 들어가면 해당 단어부터 생략
        - font_step : 크기 축소 단위
            - 크게 잡으면 빨라지지만 배치 품질이 안 좋아짐
        - prefer_horizontal : 가로 배치 시도 비율
            - 기본값 0.9
            - 1이면 세로 단어 거의 없음, 1보다 작게 설정해도 공간이 없을 때만 회전 시도한다.
    3. 단어 선택/가중치 (통계적으로 중요)
        - max_words : 표시할 최대 단어 수
        - stopwords : 불용어 설정
        - min_word_length : 최소 글자 길이
        - include_numbers : 숫자 포함 여부
        - regexp : 토큰화 정규식 
            - 정규식에 맞는 것만 클라우드에 포함
        - collocation : bigram을 하나의 구로 표시 여부
            - 기본값 : True
            - 단점 : Machine Learning과 learning이 중복 등장 가능
                - 단어 단위 분석이면 False로 하자.
        - collocation_threshold : bigram으로 인정할 임계값
            - 기본 값 : 30
        - normalize_plurals : 영어 복수형 합산
            - 기본 값 : True
        - repeat : 단어 반복하여 캔버스 채움 여부
            - 기본 값 : False
        - relative_scaling : 순위 반영할지, 빈도로 결정할지
            - 기본값 : 0.5
                - 0이면, 순위로
                - 1이면 빈도 정비례로
        - random_state : 시드 고정
    4. 색상
        - colormap : 컬러맵 (matplotlib처럼 쓰면 된다)
        - color_func : 콜백 함수로 색을 제어
    5. 마스크
        - mask : numpy 배열 전달
            - 마스크에 cloud 그려진 결과가 나온다.
4. WordCloud 주요 메서드

5. WordCloud 시각화

    ```python
    import matplotlib.pyplot as plt
    ```

6. Stop Words
    - 내장 STOPWORDS는 **영어 전용**
        - 한국어 불용어는 내장돼 있지 않으므로 직접 set을 만들거나, 형태소 분석 단계에서 조사, 어미를 걸러내고 넣어야 한다.
    - `stopwords=None`이면 내장 STOPWORDS가 자동 적용
        - 불용어 쓰기 싫으면 `stopwords=set()`을 명시해야 한다.
    - 불용어 매칭은 소문자 변환 후 이루어진다.
        - 대소문자 구분 없음.
    - `generate_from_frequencies()`로 생성할 땐 stopword 무시, 직접 해야한다.

8. 한국어 Word Cloud
    - generate로 한국어 클라우드를 생성하면, 조사/어미가 붙어서 구성됨. 아래와 같이 해야 우리가 원하는 클라우드를 볼 수 있다.
        ```python
        from konlpy.tag import Okt          # 또는 Mecab(빠름), Kkma
        from collections import Counter
        from wordcloud import WordCloud

        okt = Okt()
        nouns = okt.nouns(text)   # 명사 추출
        nouns = [n for n in nouns if len(n) > 1]  #최소길이 
        freq = Counter(nouns) # 빈도수 계산               
        wc = WordCloud(font_path='malgun.ttf',  #한글 폰트 설정
                    background_color='white',
                    width=800, height=800)
        wc.generate_from_frequencies(freq) # freq 직접 전달해 생성
        ```
        1. 명사 추출
        2. Counter
        3. generate_from_frequencies
    - 추가로, from freq를 사용하면 단순 빈도수 대신 특정 가중치를 쓸 수 있다.
