## 내린 의사결정과 판단 근거
1. **try-except 블록에 하나의 명령만 넣도록 작성했다.**
   - 근거: 과거 회사 개발 경험. 검사하려는 예외 범위를 좁히기 위함.
2. **과제 범위를 요구사항 완료 시점까지로 한정했다.**
   - 근거: 과제는 요구사항까지만 하는 것으로 이해하고 있었음.
3. **데이터 프로덕트에서 가설에 맞춰 데이터를 그리는 방식은 사용하지 않았다.**
   - 근거: 의도한 것이 아니라, 사용한 데이터가 그럴 필요가 없는 데이터였음(우연).
4. **Time Boxing 도입**
   - 근거: 저번에 너무 시간을 많이 써서 몰입해서 하고자 도입

## 관찰된 결과
- 2주차 미션 리뷰에서 다노님이 try-except 블록에 여러 문장을 넣으면 안 된다고 언급 → 내 작성 방식(1번 의사결정)이 통용 관행과 일치함을 확인.
- 다노님 피드백: 모든 조가 WordCloud에서 멈췄으며, 기대한 것은 프로덕트를 프로젝트 기간 내내 디벨롭하는 것이었음 → 2번 의사결정이 기대와 어긋났음을 확인. 과제 완료 이후 확장은 자유임을 새로 알게 됨.
- 우리 조는 "가설에 맞춰 데이터를 그렸다"는 지적을 받지 않음. 
    - 단, 층위 구분 부족 / Actionable한 결과가 바로 보이지 않음 / 절차적 공정성에 과하게 집착할 필요 없음 등의 피드백을 받음.
- 학습 소요 시간 총 6시간. 최대 소요 항목은 데이터 모델링 연습(2시간). 딥다이브로 인한 초과분은 항목별 구분이 안 되어 미확정 → 이번 주 타임박싱 실험의 비교 기준선으로 사용.

## 수업에서 배운 내용 (사실 기록)
- 동시 실행 문제: multiprocessing Queue에서 empty() 확인 후 get()을 하는 식으로 확인과 실행을 분리하면, 여러 프로세스 동시 수행 시 확인 시점과 실행 시점 사이에 상태가 달라질 수 있음(check-then-act 경쟁 상태). get_nowait()으로 확인+획득을 하나의 원자적 동작으로 처리하고 Empty 예외를 잡는 방식으로 Atomic하게 설계해야 함.
- 스크래핑: 사업성 검증 등으로 필요하며, 데이터가 API로 깔끔히 제공되지 않는 경우가 많음. 스크래핑·EDA 모두 경험량이 능력으로 이어짐.
- 고객 선택 기준: 위생 요인(Hygiene Factor)과 동기 요인(Motivation). 위생 요인은 일정 수준 이하면 탈락 사유가 되지만 아무리 높아도 선택 이유가 되지 못함. 둘을 구분해 데이터 프로덕트를 설계해야 함.
- Actionable Data: 시계열 변화가 공감되게 설계, 타겟별로 가르칠 만한 것을 추천 등 고려 사항이 많음 → 도메인 이해가 핵심.
- 익숙한 것이 최선이 아님. 선택 시점마다 찾아보고 이유가 있어야 하며, 기술이 개발된 배경(목적)을 알아야 언제 어디에 쓸지 판단 가능.
- 학습에 가장 좋은 것은 로깅(기록–회고–개선)
- 다노님께 학습 정리 시간 고민 상담 결과 : 타임박싱 — 정한 시간까지 정리하고, 안 되면 거기서 끊고 넘어가기.
