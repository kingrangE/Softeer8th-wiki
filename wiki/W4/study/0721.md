### InfraStructure

### VPC(Virture Private Cloud)
- 가상의 사설 네트워크
- 인터넷에서 접근할 수 없고, 내부적으로는 private IP로 통신
    - 가장 어려운 점 : 인터넷에서 사용할 수 있게 하는 것

- VPC는 만들면 여러 AZ에 걸쳐서 존재하도록 만든다.
    - 여러 DC를 하나의 망으로 연결해놨다.

- Subnet -> VPC 네트워크 나눠주는 그룹
    - Subnet은 VPC와 AZ 교집합에 생성된다.
        - Subnet은 AZ에 종속
    - public subnet : 우리 VPC 바깥에서 접속할 수 있는 서브넷
    - private subnet : 우리 VPC 바깥에서 접속 불가한 서브넷
        - DB나 Backend Application과 같은 외부 접속을 막아야 하는 정보를 이곳에 보관한다.
        - 기본적으로는 VPC가 내부 사설망이라 private subnet이다.
            - public subnet을 만들려면 무슨 작업을 해야 한다.
        - 여기에 DB 인스턴스를 설치하고 싶으면 어떻게 해?
            - public subnet에 NAT 인스턴스를 만들어서, NAT 인스턴스를 이용하여 바깥에 요청을 보낸다.
            - NAT Instance
                - Private Instance들이 아웃바운드 방향으로만 나갈 수 있게 해주는 EC2
                - 만드는 법
                    - NAT Instance는 공인(Public)IP가 필요하다.
                    - VPC는 사설망이기 때문에 외부와 통신할 수 있는 구멍(IGW)을 만들어준다.
                        - 이것이 있어야 이 구멍을 통해 나가서 외부와 통신
                        - IGW : Internet gateway
                    - Routing Table에 내부 IP 외에는 IGW로 보내도록 작성하기
                        - 구멍을 뚫어주기
                    - Security Group을 설정

- VPC에 들어가서 EC2를 만들었다 가정
    - Q) AZ를 선택 안했는데 그럼 랜덤으로 생성돼요?
        - ㄴㄴ EC2 선택할 때 선택한 서브넷이 AZ를 결정하도록 되어있다.
- default VPC는 쓰지마라
    - 그냥 만들어서 써라.
    - default는 연습용임
    - VPC를 만들고, private / public subnet을 각각 1개씩은 만들고 사용하자.
- VPC 만들 떄 어려운 점
    1. IP 대역을 정하라 함
        - CIDR 표기법
    2. SUBNET 지정시에도 CIDR 표기법으로 나눠주면 된다.

### Routing Table
> VPC 내부에 Route를 명시하는 기능
- VPC를 생성하면 기본적으로 Main이라고 하는 Routing Table이 생성된다.
    - Routing Table에 IP를 적으면 해당 IP는 접근 권한을 제공해준다.
    - Main Routing Table에 VPC 내부에서 통용되는 IP가 local로 존재하기 때문에 subnet끼리 기본적으로 통신이 가능하다.
- Subnet의 방화벽(NACL)을 구축하는 용도 
    - NACL은 닫지 않으면 기본적으로 ALL OPEN
        - SECURITY GROUP은 기본적으로 전부 닫혀있다. 
            - 열어줘야만 접근 가능
        - ALL OPEN인데 왜 써요?
            - Subnet에 여러 개의 EC2가 존재하면, 공통적인 설정을 할 때 사용

- Q) 우리가 Private과 Public으로 Subnet을 구분하여 이용하고 싶으면 어떻게 해야할까?
    - 조건
        1. Public은 외부에서 접근 가능해야 한다.
        2. Private은 외부에서 접근이 불가해야 한다.
        3. Private Subnet에서 외부로 요청을 보낼 수는 있어야 한다. (직접적으로 하면 안 된다)
        4. NAT Gateway를 이용하면 안 된다.
        5. VPC의 IP는 10.0.0.0/16이다.
    - 해결 방법
        1. VPC를 나눈 서브넷 중 2개가 각각 아래와 같다고 가정하겠다.
            - public : 10.0.0.0 ~ 10.0.63.255
            - private : 10.0.64.0 ~ 10.0.127.255
        2. Private은 외부에서 접근이 불가해야 하고, Public은 외부에서 접근이 가능해야 한다.
            - 따라서 IGW(Internet Gateway)를 이용해서 외부로 접속할 수 있는 통로를 만든다.
                - 이 통로를 private은 이용하지 못해야 하니, main을 수정할 수는 없고 아래와 같이 따로 만들 것이다.
                - Public Route Table
                    1. 10.0.0.0/16 -> local
                    2. 0.0.0.0 -> IGW
        3. private은 외부로 보내는 요청이 불가능하기 때문에 열린 곳을 이용해서 요청을 날려야 한다.
            - 현재 열린곳 = IGW, 따라서 IGW로 보내줘야 한다.
            - 이것을 받아주는 곳을 NAT Instance라고 해서 public에 새로 EC2를 만든다.
            - private은 main route table에 아래와 같이 수정하여 이용한다.
                1. 10.0.0.0/16 -> local
                2. 0.0.0.0 -> NAC (인스턴스 -> NAC instance 선택)
        - 그럼 Private도 외부로 요청을 보낼 수 있지만, private으로 들어오는 것은 막을 수 있고 public은 외부와 통신할 수 있다.
